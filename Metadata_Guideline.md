# Legal_DataBase 메타데이터(Metadata) 구성 가이드라인 v1.0

본 가이드라인은 Supabase의 `legal_knowledge` 테이블 내 `metadata` (JSONB) 컬럼에 어떤 정보를, 어떤 규칙으로 적재할 것인지 정의하는 문서입니다. 
향후 다양한 출처(공공 API, 엘박스, 개인 메모 등)에서 데이터를 수집할 때, 이 가이드라인을 기준으로 AI 파이프라인(파이썬 스크립트)이 동작합니다.

---

## 1. 메타데이터 설계 원칙
1. **확장성:** 정해진 스키마(표)에 얽매이지 않고, 데이터 출처에 따라 필요한 필드(Key)를 자유롭게 추가합니다.
2. **검색 최적화:** AI가 혼동하지 않도록(환각 방지), 반드시 **승패소 여부, 심급, 데이터 신뢰도** 등 객관적인 하드 팩트(Hard Fact)를 위주로 담습니다.
3. **규칙성:** 동일한 개념(예: 사건번호)은 출처가 달라도 동일한 Key 이름으로 통일합니다.

---

## 2. 필수 권장 필드 (Core Fields)
어떤 출처의 판례/지식이든 가능하면 AI를 통해 반드시 추출하여 채워 넣어야 하는 핵심 키(Key) 값들입니다. 이 필드들은 RAG(검색 증강 시스템)의 **사전 필터링(Pre-filtering)**에 집중적으로 활용됩니다.

| 필드명(Key) | 데이터 타입 | 설명 및 예시 | AI 활용 목적(RAG) |
| :--- | :--- | :--- | :--- |
| `case_type` | string | 사건의 큰 분류 (예: "민사", "형사", "가사", "행정", "조세심판") | 특정 소장 작성 시 완전히 다른 성격의 법리가 섞이는 것을 원천 차단 |
| `instance` | string | 심급 (예: "대법원", "고등법원", "1심", "특허심판원") | 하급심 판례보다 대법원 판례에 높은 가중치를 부여할 때 사용 |
| `result` | string | 재판 결과 / 승패 (예: "승소", "일부승소", "패소", "파기환송") | AI에게 "과거 패소했던 논리나 방어 실패 사례는 제외해" 라고 지시할 때 |
| `reliability` | integer(1~5) | 이 지식 데이터의 신뢰도 점수 (예: 5=대법원 전원합의체, 1=인터넷 블로그) | 여러 개가 검색됐을 때 점수가 높은 공신력 있는 자료를 우선 채택 |
| `legal_issues` | array(string) | 사건의 핵심 쟁점 키워드들 (예: `["소멸시효", "신의성실"]`) | 벡터 검색의 한계를 보완하는 명시적 쟁점 매칭 (GraphRAG 효과) |
| `defense_strategy`| string | 승소를 이끌어낸 주력 논리 유형 (예: "절차적 하자", "사실관계 오인") | AI가 초안 작성 시 "유사 사건 승소 전략"을 근거로 제시하기 위함 |
| `vector_version` | string | 임베딩 생성 AI 모델 (예: `"gemini-embedding-001"`) | 향후 임베딩 모델 업그레이드 시 구/신버전 충돌 방지 시스템 필드 |

---

## 3. 출처별 맞춤형 선택 필드 (Custom Fields)
데이터를 가져온 곳(Source)의 특성에 맞춰, 파이프라인 파이썬 스크립트가 알아서 붙여줄 선택적 꼬리표들입니다.

### A. 일반 법원 판례 (대법원, 엘박스 크롤링 등)
*   `judge_name`: (string) 담당 재판장 이름 (특정 판사의 판결 성향 분석용)
*   `judgment_date`: (YYYY-MM-DD) 선고 일자 (최신 판례 우선 적용 로직용)
*   `related_articles`: (array) 주요 참조 조문 (예: `["민법 제750조", "민법 제390조"]`)

### B. 행정기관 결정문 (조세심판원, 공정위 등 특수직역)
*   `agency`: (string) 처분청 또는 재결청 (예: "조세심판원", "금융감독원")
*   `claim_amount`: (number) 청구 세액 또는 과징금 규모
*   `dispute_type`: (string) 분쟁 내역 (예: "부가가치세 부과처분 취소", "하도급대금 미지급")

### C. 사내 지식 (Zettelkasten 단축키, 회의록, 과거 서면)
*   `author`: (string) 이 메모/법리를 작성한 변호사 이름 (예: "안셀로트")
*   `project_id`: (string) 어떤 실제 사건(고객)을 수행하면서 남긴 메모인지 사건 고유번호 매핑
*   `client_name`: (string) 해당 사건 의뢰인명

## 4. 메타데이터 JSON 형태 예시 (Data Example)

향후 AI 파이프라인(Phase 2) 코딩 시, 제미나이가 원문을 읽고 아래와 같은 완벽한 JSON을 구성하여 Supabase로 던지도록(Insert) 프롬프트를 설계합니다.

```json
{
  "case_type": "민사",
  "instance": "대법원",
  "result": "파기환송",
  "reliability": 5,
  "legal_issues": ["상사유치권", "점유의 적법성"],
  "defense_strategy": "사실관계 오인 입증 (채권 발생 시점 반박)",
  "vector_version": "gemini-embedding-001",
  "judgment_date": "2023-10-15",
  "related_articles": ["민법 제398조", "하도급법 제14조"],
  "judge_name": "김대법",
  "ai_confidence": 0.98    // 파이프라인 AI가 이 요약을 확신하는 정도 (필터링 컷오프용)
}
```
